AI Agent — Python 기반 Multi-Agent 개발

PYTHON · LANGCHAIN · MULTI-AGENT ARCHITECTURE
📐 핵심 목표

단일 에이전트의 Context Overload로 인한 환각(Hallucination) 현상을 구조적으로 방지하고, VDI 도메인에 특화된 사용자·관리자 지원 시나리오를 구현했습니다. LangChain 기반의 유연한 멀티 LLM 연동 환경도 함께 구축했습니다.

1-1. Multi-Agent 구조 설계
Problem
Context Overload

단일 에이전트가 모든 VDI 도메인 지식·도구·히스토리를 하나의 Context에 담을 경우, 토큰 한계 초과 및 환각(Hallucination) 현상이 빈번하게 발생합니다.
Solution
Main-Sub Agent 분리

Main Agent가 의도를 분류하고 Sub-Agent에 위임하는 역할 분리 구조로 설계했습니다. 각 Sub-Agent는 자신의 도메인에 필요한 최소한의 Context만 보유하므로, 응답의 정확도와 일관성이 크게 향상되었습니다.
Step 01 · WebSocket / FastAPI
사용자 질문 수신 — 실시간 채팅 인터페이스
FastAPI WebSocket으로 낮은 지연(Low-latency) 실시간 응답을 제공합니다.
Step 02 · Main Agent
의도 분류 및 Sub-Agent 라우팅
질문 의도를 사용자 지원 관리자 지원 일반 문의 로 분류 후, 담당 Sub-Agent로 작업을 위임합니다.
Step 03 · Memory Manager
대화 히스토리 기반 Context 유지
대화 이력(Conversation History)을 참조해 멀티턴(Multi-turn) 대화에서도 문맥이 끊기지 않도록 관리합니다.
Step 04 · Sub-Agents (Domain Specialized)
VDI 도메인 특화 작업 실행
사용자 지원 에이전트: VM 상태 이슈 자가 진단 및 조치 방법 안내, 사용자 자원 현황 조회.
관리자 지원 에이전트: 다수 사용자의 VDI 이력 추적, 상태 일괄 업데이트 등 반복 관리 업무 자동화.
Step 05 · LangChain · LLM Router
동적 LLM 선택 및 호출
프로젝트 특성·비용·성능 요구에 따라 아래 모델 중 최적 모델로 동적 교체가 가능합니다.
🔌 연동 LLM 목록 (LangChain 기반)
OpenAI GPT Anthropic Claude Gemini Grok Luxia Friendli Ollama (Local)

LangChain의 추상화 레이어를 활용해 모델 교체 시 비즈니스 로직 변경 없이 config만으로 전환 가능하도록 구현했습니다.

1-2. VDI 도메인 특화 프롬프트 엔지니어링
🎯
정교한 도메인 지식 임베딩
VMware VDI 환경(DataCenter, Cluster, Host, VM 계층 구조, 이벤트 유형)에 대한 도메인 지식을 시스템 프롬프트에 체계적으로 작성하여, 에이전트가 가상화 환경 특유의 이슈를 정확하게 해석하고 응답할 수 있도록 고도화했습니다.
👤
사용자 지원 에이전트 시나리오
VM 상태 이슈(연결 불가, 성능 저하 등)를 AI가 자동 진단하고 자가 조치 가이드를 제공합니다. 사용자가 직접 자신의 VDI 자원 현황을 조회하고 관리할 수 있는 대화형 인터페이스를 구현했습니다.
🛠
관리자 지원 에이전트 시나리오
수십~수백 사용자의 VDI 이력을 추적하고, 상태 업데이트·이상 감지 등 반복적인 관리 업무를 자동화하는 시나리오를 설계했습니다. 관리자 생산성을 높이고 운영 오류를 줄이는 것이 핵심 목표입니다.
🧠 환각 방지 전략

역할 분리로 각 에이전트의 Context 범위를 최소화. 필요한 도구·지식만 주입해 응답 정확도를 높였습니다.

🔌 LLM 유연성

7종 이상 LLM을 config 변경만으로 교체 가능. 프로젝트 예산·성능 요건에 즉각 대응합니다.

📋 자가 진단 UX

사용자가 IT 지식 없이도 VM 이슈를 AI와 대화하며 스스로 해결할 수 있는 경험을 제공합니다.

⚙️ 관리 자동화

반복적인 VDI 상태 추적·업데이트 업무를 에이전트가 자동 처리하여 관리자 운영 부담을 줄였습니다.

vCompute — C# 기반 vCenter 리소스 수집 엔진

C# · VMWARE VIM SDK · MEMORY CACHING
📐 핵심 목표

.NET Framework 4.5.2와 VMware Vim SDK를 활용해 vCenter 계층 구조(DataCenter → Cluster → Host → VM)의 리소스 정보와 이벤트·태스크 이력을 조회하는 REST API를 개발했습니다. SDK의 페이징 부재로 인한 서버 과부하·응답 지연·먹통 현상을 서버 측 Count 제한과 호출 모듈의 커서 기반 순차 재조회 설계로 해결하여 대규모 인프라 환경에서도 안정적인 성능을 확보했습니다.

2-1. VMware Vim SDK 연동 및 REST API 개발
🔌
vCenter 이벤트·태스크 수집 API
.NET Framework 4.5.2 + VMware.Vim SDK를 활용해 DataCenter, Cluster, Host, VM의 리소스 현황과 이벤트(Event) 및 태스크(Task) 이력을 조회하는 REST API를 설계·개발했습니다. API는 startTime(시작시간)과 id(선택)를 파라미터로 수신합니다.
2-2. 커서 기반 순차 재조회 설계
Problem
SDK 페이징 부재 · 대량 응답 문제

VMware Vim SDK에는 표준 Paging 기능이 없어, 한 번의 호출로 수만 건이 반환될 수 있습니다. 이를 제한 없이 허용하면 서버 과부하 · 응답 지연 · 먹통 현상이 발생합니다.
Solution
서버 Count 제한 + 호출 모듈 순차 재조회

엔진 서버는 web.config의 최대 조회 건수(Count)만큼만 반환하고, 외부 호출 모듈이 반환된 마지막 이벤트/태스크의 시작시간과 ID를 커서로 삼아 다음 구간을 재조회하는 방식으로 전체 데이터를 순차 수집합니다.
📋
API 파라미터 분기 — startTime + id (선택)
호출 모듈은 두 가지 파라미터를 엔진 API에 전달합니다.

① id 없이 호출 — startTime부터의 이벤트·태스크를 Count 건만큼 반환합니다. 최초 조회 또는 새 구간 시작 시 사용합니다.

② id 포함 호출 — startTime 범위 내에서 해당 id 다음 레코드부터 조회합니다. 동일 타임스탬프 구간 내 누락 없이 이어서 수집할 때 사용합니다.
Step 01 · 호출 모듈 → 최초 API 요청
startTime 전달, id 없이 첫 조회
호출 모듈이 수집 시작 시간(startTime)만 전달합니다. 엔진은 해당 시점부터 web.config의 Count만큼 이벤트·태스크를 조회하여 반환합니다.
Step 02 · 엔진 → Count 단위 반환
최대 조회 건수만큼 이벤트·태스크 응답
서버는 Count 건을 채워 응답합니다. 응답에는 마지막 레코드의 이벤트/태스크 시작시간과 고유 ID가 포함됩니다.
Step 03 · 호출 모듈 → 잔여 데이터 판단
반환 건수 == Count 이면 추가 데이터 존재로 판단
반환 건수가 Count와 같으면 호출 모듈은 잔여 데이터가 있다고 간주하고 재조회를 준비합니다. 반환 건수 < Count이면 수집 완료로 판단하고 종료합니다.
Step 04 · 호출 모듈 → 커서로 재조회
마지막 시작시간 + id를 커서로 API 재호출
호출 모듈이 직전 응답의 마지막 이벤트/태스크 시작시간과 id를 파라미터로 엔진 API를 재호출합니다. 엔진은 해당 id 다음 레코드부터 Count만큼 반환하여 중복·누락 없는 연속 수집을 보장합니다.
Step 05 · 반복 → 수집 완료
반환 건수 < Count 될 때까지 Step 02–04 반복
호출 모듈이 Step 02–04를 반복하며 전체 이벤트·태스크를 순차 수집합니다. 마지막 페이지에서 반환 건수가 Count보다 적으면 수집을 종료합니다.
2-3. 세션 버퍼 기반 메모리 캐싱 전략
⚡
원격 호출 최소화 → 즉시 메모리 반환
vCenter API의 네트워크 지연(RTT)과 서버 부하를 줄이기 위해 조회된 데이터를 세션 버퍼에 캐싱합니다. 이후 동일 요청은 원격 호출 없이 메모리에서 즉시 반환하여 대규모 인프라 환경에서도 안정적인 API 서비스를 제공합니다.
🔁 호출 모듈 재조회 설계

재조회 책임을 외부 호출 모듈에 위임하여 엔진 서버 부하를 최소화. 엔진은 단순 Count 반환에 집중합니다.

⚙️ 서버 Count 제한

web.config의 Count 값으로 최대 응답 건수를 제어. 인프라 환경에 맞게 튜닝 가능합니다.

🔒 중복·누락 방지

시작시간 + ID 커서 방식으로 페이지 경계에서의 이벤트·태스크 중복·누락을 구조적으로 방지합니다.

⚡ 캐싱 TTL

moREF 키 기반 10초 TTL로 중복 SDK 호출을 차단, 응답 속도 개선 및 vCenter 서버 보호.

Java / Spring Boot 기반 통합 API 서버

JAVA · SPRING BOOT · MCP · RETROFIT
📐 핵심 목표

사용자 Web 요청(REST)과 Python AI Agent의 MCP Tool 호출이라는 두 인터페이스를 단일 Service 레이어로 통합했습니다. 또한 이종 언어 시스템인 C# vCompute를 Retrofit으로 타입 안정성 있게 연동하여 Java 생태계 안에서 일관된 방식으로 인프라 데이터를 활용할 수 있도록 했습니다.

3-1. 하이브리드 인터페이스 — Service Layer 공통화
🔒
Spring Security (JWT / SSO)
모든 요청의 관문 — 인증 · 인가 · 필터 체인
↓ 인증된 요청만 통과
🌐
REST Controller
@GetMapping · @PostMapping
← 사용자 Web 요청 수신
🤖
MCP Tool
@Tool · @ToolParam
← Python AI Agent 전용
↓ 동일한 Service 메서드 호출
⚙️
Unified Service Layer
REST · MCP 양쪽 인터페이스에서 공용으로 사용하는 단일 비즈니스 로직.
인터페이스가 추가돼도 Service 코드 변경 없이 확장 가능.
↓ 하위 시스템 호출
⚙️
Retrofit (REST) ↓
vCompute Connector
C# vCompute REST API 호출 · 타입 안전(Type-safe) 연동
3-2. Retrofit을 활용한 이종 언어 시스템 연동 (Java ↔ C#)
🔗

Type-safe Cross-language Integration

Retrofit 라이브러리를 도입하여 C#으로 작성된 vCompute의 REST API를 Java 환경에서 타입 안정성(Type-safe)있게 호출합니다. 인터페이스 선언만으로 HTTP 통신 코드가 자동 생성되므로, 보일러플레이트 없이 일관된 방식으로 이종 언어 시스템을 연동할 수 있습니다.

📋
Retrofit 인터페이스 선언 방식
Java 인터페이스에 @GET, @POST 어노테이션으로 C# API 엔드포인트를 선언합니다. Retrofit이 런타임에 구현체를 자동 생성하므로, C# API 명세가 바뀌어도 인터페이스 수정만으로 대응 가능합니다.
⚡
타입 안정성 확보 → 런타임 오류 사전 방지
응답 데이터를 Java DTO(Data Transfer Object)로 자동 역직렬화합니다. 컴파일 단계에서 타입 불일치를 감지하여 이종 언어 간 통신에서 발생하기 쉬운 런타임 오류를 사전에 방지합니다.
🔌 이중 인터페이스

REST · MCP 두 프로토콜을 동시에 지원하면서도 단일 Service 코드로 관리합니다.

🔗 Retrofit 연동

C# vCompute API를 Java에서 타입 안전하게 호출. 보일러플레이트 코드 없이 이종 언어 통합을 실현합니다.

📦 확장성

새 인터페이스 추가 시 Service 레이어 변경 없이 Controller/Tool 레이어만 추가하면 됩니다.

🔒 통합 보안

JWT·SSO 인증이 단일 Spring Security 필터 체인에서 처리되어 모든 인터페이스에 일관되게 적용됩니다.
