보안 모듈 설계 - 네트워크 암호화 & 무결성 검증

VDI 포털 시스템에서 사용자 PC(C# Launcher)와 Java 웹 서버 간의 전송 구간을 보호하기 위해 직접 설계·구현한 보안 아키텍처입니다. RSA-AES256 하이브리드 방식으로 VM 접속 정보를 암호화하고, 보안 프로그램의 무결성을 3단계로 순차 검증합니다.

RSA-2048 AES-256 JWT Auth Custom Protocol C# .NET Java Spring SHA-256 Hash Key Wrapping
01

RSA-AES 하이브리드 암호화 시퀀스

네트워크 전송 구간 보호 — Web Server ↔ C# Launcher

설계 목적
VDI 접속 시 사용자 PC가 VM 접속 정보를 수신하는 네트워크 구간은 스니핑 공격에 노출될 수 있습니다. HTTPS 외에 추가적인 페이로드 암호화 레이어를 적용하여, 오직 해당 클라이언트의 개인키를 가진 쪽에서만 복호화할 수 있도록 설계했습니다. 대칭키(AES)의 빠른 암호화 속도와 비대칭키(RSA)의 안전한 키 전달을 결합한 하이브리드 구조입니다.
Phase 1 · Launch Trigger
Launcher 기동 및 JWT 전달
웹 서버는 .slv 파일 또는 커스텀 프로토콜을 통해 C# Launcher(.exe)를 실행시킵니다. 이 과정에서 callbackUrl이 포함된 JWT를 함께 전달하며, Launcher는 JWT를 디코딩하여 콜백 주소를 추출합니다.
.slv 파일 연결 또는 커스텀 URI 스킴을 통해 OS 레벨에서 Launcher가 자동 실행됩니다.
Phase 2 · Callback & Key Exchange
RSA 키 생성 및 Callback 요청
Launcher는 호출 시마다 1회용 RSA 키 쌍(공개키/개인키)을 새로 생성합니다. 이후 추출한 callbackUrl로 HTTP 요청을 보내며, 이때 수신받은 JWT(서버 인증용)와 RSA 공개키를 헤더에 함께 담아 전송합니다.
Header: Authorization: Bearer <JWT> + RSA-Public-Key
Phase 3 · Server-side Processing
서버 측 암호화 처리 (AES-256 + Key Wrapping)
서버는 요청 헤더의 JWT를 검증(서명·만료·인증)한 후, 랜덤 AES-256 대칭키를 생성합니다. VM 접속 정보와 프로세스 체크 정보를 AES키로 1차 암호화하고, 생성된 AES키 자체는 클라이언트의 RSA 공개키(RSA-OAEP)로 2차 암호화(Key Wrapping)합니다.
Response Header: 암호화된 AES키 (RSA-OAEP)  |  Response Body: 암호화된 JSON (AES-256-CBC)
Phase 4 · Client Decryption
클라이언트 복호화 및 VM 접속
Launcher는 자신이 보유한 RSA 개인키로 헤더의 암호화된 AES키를 먼저 복호화합니다. 이후 복원된 AES키로 Body의 JSON 데이터를 복호화하여 VM 접속 정보를 획득하고 접속을 실행합니다.
🔑
핵심 포인트
RSA 키 쌍을 요청마다 새로 생성하는 구조 덕분에, 이전 세션에서 키가 유출되더라도 다른 세션에는 전혀 영향이 없습니다. 또한 AES키가 클라이언트의 RSA 공개키로 래핑되어 전송되므로, 해당 클라이언트의 RSA 개인키 없이는 어떤 경우에도 복호화가 불가능합니다.
02

다단계 보안 프로그램 무결성 검증 플로우

엔드포인트(사용자 PC) 보안 제어 — 순차 3단계 검증

설계 목적
VDI에 접속하는 사용자 PC에 필수 보안 프로그램이 실행 중인지, 정상 버전인지, 파일이 위변조되지 않았는지를 순차적으로 검증하여 보안 기준을 충족하지 못한 단말의 접근을 차단합니다. 각 단계는 독립적으로 에러를 반환하도록 설계되어 운영 시 정확한 원인 파악이 가능합니다.
STEP 1
🔍
실행 여부 체크 Execution Check
서버에서 전달받은 running_process_infoes 리스트의 ProcessName과 로컬 PC에서 실행 중인 프로세스 목록을 스캔하여 대조합니다. 보안 프로그램이 실행되지 않은 경우 이후 단계로 진행하지 않고 즉시 차단합니다.
⛔ Error 1 — 프로세스 미실행 · 접속 차단
STEP 2
📋
버전 체크 Version Match
실행 중인 보안 프로그램의 버전 정보를 서버에 등록된 허가 버전 목록과 비교합니다. 버전이 일치하지 않으면 사용자에게 업데이트를 안내하고 접속을 중단합니다. 구버전 보안 프로그램으로 인한 취약점 노출을 방지합니다.
⛔ Error 2 — 버전 불일치 · 업데이트 안내
STEP 3
🔏
무결성 체크 Hash Verification
실행 파일의 바이너리를 실시간으로 읽어 SHA-256 해시를 추출하고, 서버에 저장된 공식 해시값과 비교합니다. 해시가 다를 경우 파일이 위·변조된 것으로 판단하여 접속을 즉시 차단합니다. 보안 프로그램 바이너리가 외부에서 교체·패치되는 공격을 탐지하기 위한 방어선입니다.
⛔ Error 3 — 위변조 감지 · 접속 차단
FINAL
✅
최종 승인 — VM 접속 실행 Approved
3단계 검증을 모두 통과한 경우에만 복호화된 vm_Connection_Info를 사용하여 VM에 접속합니다. 검증 실패 시 어느 단계에서도 이후 프로세스로 진행되지 않으며, 각 에러 코드로 운영팀이 원인을 즉시 파악할 수 있습니다.
🛡️
핵심 포인트
단순히 프로그램 설치 여부가 아닌 실행 중 → 버전 일치 → 파일 무결성의 3단계를 모두 통과해야만 접속이 허용됩니다. 특히 Hash 검증은 보안 프로그램이 정상 버전이더라도 바이너리가 임의로 수정·교체된 경우를 탐지하여, 우회 공격까지 방어할 수 있는 구조입니다.
