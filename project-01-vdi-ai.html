01

AI Agent — 지능형 자원 관리 시스템 구축

PYTHON · LANGCHAIN · CONTEXT MANAGEMENT
🎯 설계 핵심

AI Agent Admin Console 플랫폼 위에서 VDI 도메인 특화 에이전트를 구축했습니다. LangChain create_agent(ReAct)로 에이전트를 구성하며, Main Agent(오케스트레이터)가 독립 에이전트인 Sub-Agent들을 Tool로 바인딩하는 Multi-Agent 구조를 채택했습니다. LangChain이 각 LLM ChatModel을 감싸는 래퍼를 통해 병렬 Tool 호출을 Config 하나로 제어하고, 장기 대화의 Context Window 한계를 모델별 컨텍스트 레지스트리 및 Auto-Compact (이력 자동 압축)으로 극복했습니다.

1-0. Multi-Agent 아키텍처 — Main(오케스트레이터) + Sub-Agents
🤖
독립 에이전트를 Tool로 바인딩하는 Multi-Agent 구조
Admin Console에서 에이전트를 생성할 때 type: main 또는 type: sub를 설정합니다. Sub-Agent는 Main과 동일한 독립 에이전트이며, 단지 Main Agent의 Tool 목록에 바인딩되어 LangChain ReAct 루프 안에서 호출됩니다.

VDI AI 에이전트는 Main(VDI 오케스트레이터)이 사용자 의도를 파악하고, VM 상태 분석·자원 보고서·인프라 진단·향후 조치 등을 담당하는 Sub-Agent들에게 위임하는 방식으로 역할을 분산했습니다.
Why
단일 Agent의 컨텍스트·역할 과부하

VDI 관련 요청은 자원 조회, 장애 진단, 보고서 생성, 인프라 조치 등 성격이 다른 태스크가 혼재합니다. 단일 Agent에 모든 Tool과 지시를 넣으면 컨텍스트가 폭증하고 LLM의 선택 정밀도가 저하됩니다.
✓ 해결
역할별 Sub-Agent 분리 → Tool 바인딩

도메인별로 독립 Sub-Agent를 별도 생성하고 Main Agent의 Tool로 등록했습니다. Main은 사용자 의도만 파악해 적절한 Sub-Agent를 호출하고, Sub-Agent는 자신의 역할에 특화된 Tool과 시스템 프롬프트로 깊이 있는 처리를 수행합니다. 각 Sub-Agent도 Admin Console에서 독립적으로 관리·버전 관리됩니다.
1-1. LangChain ChatModel 래퍼의 병렬 Tool 호출 파라미터 제어
⚡
LangChain ChatModel 래퍼 기반 병렬 호출 제어
LangChain은 GPT·Claude 등 각 LLM 공급사의 API를 ChatOpenAI, ChatAnthropic 등의 ChatModel 클래스로 감쌉니다. 이 래퍼는 모델이 지원하는 경우 병렬 Tool 호출 파라미터를 직접 노출합니다 — 하나의 LLM 추론 패스에서 여러 Tool을 동시에 호출하는 방식입니다. 이를 Config 값으로 제어해 직렬(Serial) ↔ 병렬(Parallel) 호출 모드를 전환할 수 있도록 구현했습니다.
Issue
LLM 공급사별 상이한 파라미터 규격

병렬 호출 파라미터 명칭이 모델마다 달랐습니다. GPT 계열 ChatModel은 parallel_tool_calls=True, Claude 계열은 반전 논리인 disable_parallel_tool_use=False 구조를 사용합니다. 공급사가 바뀔 때마다 호출부 코드를 수정해야 하는 문제가 있었습니다.
Solution
Model Factory 패턴으로 파라미터 추상화

model_factory.py에서 LangChain ChatModel 인스턴스를 생성할 때, 환경 변수 PARALLEL_TOOL_CALLS 하나만 읽어 공급사별 실제 파라미터명으로 자동 매핑합니다. 호출부는 모델 공급사에 무관하게 동일한 Config 값으로 직렬·병렬 모드를 제어할 수 있습니다.
📊
직렬(Serial) vs 병렬(Parallel) — 호출 방식 비교 및 효과
직렬 호출(Serial): LLM이 Tool A 결과 반환 → Tool B 호출 → Tool C 호출 순으로 각 Tool 결과를 받은 뒤 다음 Tool을 선택합니다. 독립적인 작업(VM 조회 + DB 쿼리 + 상태 진단)도 반드시 순서대로 실행되어 총 대기 시간이 N배 누적됩니다.

병렬 호출(Parallel): LLM이 단일 추론 패스(Single Inference Pass)에서 독립적인 Tool 여러 개를 동시에 호출 목록으로 묶어 emit합니다. 모든 Tool 결과가 수집된 뒤 한 번에 LLM에 반환되므로, N개 Tool의 총 대기 시간이 가장 느린 Tool 1개의 응답 시간 수준으로 단축됩니다.

적용 효과  VM 상태 조회·자원 통계·로그 수집 3개 Tool을 동시 호출하는 시나리오에서, 직렬 대비 체감 응답 속도가 약 2~3배 단축되었으며, 사용자 화면에 AI 결과가 표시되기까지의 대기 시간이 대폭 감소했습니다.
1-2. 장기 대화 지속을 위한 Auto-Compact (히스토리 자동 압축)
📦
설계 의도: 토큰 비용 절감 및 환각 차단
대화가 길어질수록 모델 비용이 폭증하고, 허용 토큰 초과 시 에러가 발생하는 문제를 해결하기 위한 자체 압축 파이프라인.
Phase 1 / Monitor
4단계 독립 영역 관측 (Observer Pattern)
하나의 컨텍스트를 [시스템 프롬프트, 도구 스키마, 이전 대화 이력, 현재 요청]의 네 가지 섹터로 분리해 토큰 소모량을 개별 모니터링하며, 합산 사용량이 임계치(Threshold)에 도달하는지 관측·로깅합니다.
Phase 2 / Compactor
LLM 기반 요약 압축 (단일 Summary 본 생성)
설정된 최근 대화 수(`MIN_PRESERVE_TURNS`)는 원본으로 남겨두고 나머지를 LLM을 통해 하나의 짧은 요약(Summary) 버퍼로 묶습니다. 첫 압축(`FULL`) 이후로는 기존 요약과 추가된 기록을 누적(`PARTIAL`)하여 하나의 짧은 스냅샷 텍스트로 보존합니다.
Fallback Strategy
LLM 장애 시 FIFO 트리밍 방어
LLM 서버 오류로 압축이 실패하더라도 서비스가 멈추지 않도록, 오래된 대화를 하나씩 폐기하는 선입선출(FIFO) 트리밍 기법이 자동 트리거되도록 예외 처리 시스템인 Fallback 방어로직을 짰습니다.
1-3. 안전 마진 기반의 컨텍스트 차단 시스템
🛡
Model Context Registry 구현
LLM 모델 및 엔진별로 모두 제각각인 max_input_tokens 규격을 하드코딩한 Registry를 구성했습니다. 이를 바탕으로 `전체 Input 한계` - `초기 할당 Output 한계` 계산 식을 거쳐 최종 사용 가능한 안전 버퍼 토큰(Usable Input Tokens)을 산출합니다. LLM API 호출 직전 계산된 입력 양이 이 버퍼를 초과하면 사전(Pre-flight) 단계에서 차단하여 API 에러 비용을 없앴습니다.

1-4. Sub-Instruction Tool — 컨텍스트 선택적 로드
📋
필요한 지침만 그때그때 검색·로드
에이전트의 지침이 방대해질수록 모든 내용을 시스템 프롬프트에 상시 포함하면 매 요청마다 토큰 낭비가 발생합니다. 이를 해결하기 위해 에이전트가 필요한 시점에만 Sub-Instruction을 검색·로드할 수 있는 별도 Tool을 개발했습니다. 전체 지침을 컨텍스트에 상시 포함하지 않는 선택적 로드 방식으로, Tool Search와 함께 컨텍스트 효율성을 이중으로 확보합니다.

vCompute — C# 기반 vCenter Event 엔진

C# · .NET SDK · CURSOR-BASED PAGINATION
Problem
SDK 페이징 부재로 인한 먹통 이슈

VMware의 Vim API(C#)를 사용해 대규모 인프라 로그를 조회할 때 기능상 Paging 속성이 미지원 상태였습니다. 이로 인해 한 번에 수만 건의 쿼리가 발생해 타임아웃 지연과 먹통 현상이 생겼습니다.
Solution
커서 기반 페이징 (시작 시간 + ID 방식)

서버 사이드(vCompute)에서는 Config로 최대 반환 개수만 하드 제어하고, 호출 모듈(Java) 측에서 반복적으로 [마지막 기록 시간 (StartTime)]과 마지막 개체의 [고유 ID]를 다음 데이터의 Cursor 파라미터로 삼아 순차적으로 이어서 API를 재귀호출하도록 분산 처리 설계를 적용했습니다.
⚡
메모리 캐싱 (Session Buffer) 활용
VMware vCenter로의 원격 호출은 느리고 비쌉니다. moREF 식별키를 기반으로, 동일 조회 객체는 10초의 버퍼 시간을 두어 TTL을 설정, 다중 스레드의 병렬 조회 폭주로 vCenter가 다운되는 것을 막았습니다.
