🔌
외부 MCP 서버 연동 — SSE / stdio
SSE(Server-Sent Events)나 stdio 방식의 외부 MCP 서버를 UI에서 등록하고 Agent에 연결할 수 있습니다. VDI·vCenter 데이터를 제공하는 Java/Spring AI MCP 서버(SSE)도 이 연동을 통해 Agent에 연결됩니다. defer_loading: true로 선택적 Tool 로드도 지원합니다.
📋
Sub-Instruction Tool — 지침 선택적 로드
방대한 에이전트 지침을 시스템 프롬프트에 상시 포함하면 매 요청마다 토큰 낭비가 발생합니다. 에이전트가 필요한 시점에만 Sub-Instruction을 검색·로드하는 Tool을 개발하여, Tool Search와 함께 컨텍스트 효율성을 이중으로 확보했습니다.
🧪
TypeScript Dynamic Data Analyzer MCP Tool Server

배경 및 문제 인식

CONTEXT · PROBLEM STATEMENT
🎯 프로젝트 목적

AI Agent Builder는 개발자가 코드를 직접 작성하지 않고도 LangChain 기반 Agent를 구성·배포할 수 있는 자체 플랫폼입니다. Agent에 연결되는 Tool의 수가 증가함에 따라 — 특히 외부 MCP 서버의 도구들까지 포함하면 수백 개에 달하는 Tool Schema가 매 요청마다 LLM 컨텍스트에 전달되어 입력 토큰이 폭증하는 문제가 발생했습니다. 이를 해결하기 위해 GPT·Claude 양측의 Tool Search 기능을 도입했습니다.

Issue
Tool Schema 과부하

MCP 서버 3개 이상 연결 시 Tool definition이 context window의 30~40%를 차지. 외부 MCP 서버의 도구는 클라이언트 단에서 Tool Schema를 직접 확인할 수 없고, LLM 서버 내부에서 처리되기 때문에 전체 Tool 목록을 매번 주입하는 방식은 비용·성능 양 측면에서 심각한 병목이 되었습니다.
✓ 해결
GPT 계열은 {type: "toolsearch"} 파라미터로, Claude 계열은 tool_search_tool_bm25 / tool_search_tool_regex 중 사용자가 모드를 선택하는 방식으로 Tool Search를 활성화. 모든 외부 MCP 도구에 defer_loading: true를 적용하여 실제 필요한 시점에만 동적 로딩하도록 구성했습니다.
📊 Claude API Log 실측 결과
Tool Search 활성화 후 Claude API 로그 탭에서 확인한 결과, 동일 요청 기준 입력 토큰이 기존 대비 약 1/10 수준으로 감소했습니다. GPT 계열은 공식 메트릭 노출이 없어 직접 비교는 어렵지만, Tool 컨텍스트 길이 단축 효과는 동일하게 확인되었습니다.
01

Tool Search 동작 원리

MECHANISM · COMPARISON · GPT vs CLAUDE
💡 핵심 아이디어

기존 방식은 모든 Tool Schema를 매 API 호출마다 tools 배열에 담아 전송합니다. Tool Search는 이를 역전시켜, LLM이 먼저 "어떤 Tool이 필요한지"를 검색하고 — 필요한 Tool만 컨텍스트에 로드하는 Lazy Loading 전략입니다. Tool Schema는 LLM 서버 내부 인덱스에 사전 등록되며, 클라이언트는 Schema 전체를 노출하지 않아도 됩니다.

1-1. GPT vs Claude — 방식 비교
구분 GPT (gpt-5.4+) Claude (Opus / Sonnet 4.5+ / Haiku 4.5)
활성화 방법 tools 배열에 {type: "toolsearch"} 항목 추가. Model이 자체적으로 호스팅 검색 수행 tools에 tool_search_tool_bm25 또는 tool_search_tool_regex 추가 후 사용자 선택 모드로 동작
검색 알고리즘 OpenAI 내부 Vector / Embedding 기반 유사도 검색 (공개 미완료) BM25 (자연어 쿼리 의미 기반 랭킹) 또는 Regex (패턴 매칭) 선택 가능
defer_loading defer_loading: true로 개별 Tool 지연 등록. Namespace 단위 권장 defer_loading: true로 MCP 서버 전체 또는 개별 Tool 지연 등록 가능
클라이언트 실행 모드 execution: "client" 설정 시 tool_search_call emit → 앱이 검색 수행 후 결과 반환 기본적으로 서버 사이드 실행. 클라이언트 개입 없이 LLM이 tool search 수행
토큰 절감 효과 Tool 수에 따라 60~85% 감소 (OpenAI 공식 수치) 실측 기준 입력 토큰 약 1/10 수준으로 감소 (Claude API 로그 확인)
MCP 서버 연동 MCP Namespace 단위로 defer_loading 적용 권장 mcp_servers 배열에 default_config: {defer_loading: true} 설정으로 서버 전체 지연 적용
1-2. Claude — BM25 vs Regex 선택 전략
🔍
BM25 — 자연어 시맨틱 검색
tool_search_tool_bm25_20251119 변형. Okapi BM25 랭킹 함수를 사용하여 Tool 이름·설명·파라미터 명칭에 대한 자연어 Query 유사도를 계산합니다. "VM 상태 조회해줘"처럼 Tool 이름을 정확히 모르는 경우에 적합하며, 100개 이상의 대형 Tool 카탈로그에서 탁월한 성능을 보입니다. AI Agent Builder에서 기본(Default) 모드로 설정됩니다.
⚡
Regex — 패턴 기반 정밀 매칭
tool_search_tool_regex 변형. Python 정규표현식으로 Tool 이름을 직접 패턴 매칭합니다. Tool 명명 규칙이 구조적(vdi_get_*, vm_action_* 등)이고 정확한 이름을 알 때 가장 빠릅니다. 단, 복잡한 Query에서 Character Limit 오류 가능성이 있어 소규모 Tool 셋 또는 내부 명명 체계가 명확할 때 사용자 선택 옵션으로 제공합니다.
⚠️ 구현 시 유의사항
외부 MCP 서버의 Tool들은 클라이언트 단에서 Schema를 직접 열람할 수 없습니다. 따라서 Tool Search가 올바른 Tool을 탐색할 수 있도록, MCP 서버의 Tool 등록 단계에서 설명(description)과 파라미터 명칭을 최대한 상세하게 작성하는 것이 핵심 전제 조건입니다. 이 메타데이터가 BM25 또는 Regex 검색의 인덱스로 활용됩니다.

03

구현 상세 — AI Agent Builder 통합

IMPLEMENTATION · LANGCHAIN · MCP INTEGRATION
🔧 구현 전략

LangChain의 create_agent 기반으로 Agent를 구성하되, Tool 등록 단계에서 GPT / Claude 모델별 Tool Search 파라미터를 자동 분기하여 주입합니다. Agent Builder UI에서는 모델 선택 시 Claude 계열인 경우 BM25/Regex 모드 선택 UI가 동적으로 노출됩니다.

3-1. 구현 단계
Phase 1 · Tool Registry
MCP 서버 Tool 등록 및 defer_loading 설정
Agent Builder에서 MCP 서버를 연결할 때, 모든 외부 Tool에 defer_loading: true를 기본 적용합니다. 사용 빈도가 높은 핵심 Tool (상위 3~5개)만 defer_loading: false 예외 처리하여 검색 Round-trip 지연 없이 즉시 사용 가능하도록 구성합니다.
Phase 2 · Model Branching
GPT / Claude 모델별 Tool Search 파라미터 분기
GPT 계열: tools 배열에 {type: "toolsearch"} 항목 자동 삽입.
Claude 계열: Builder UI의 Search Mode 설정(BM25 or Regex)에 따라 tool_search_tool_bm25_20251119 또는 tool_search_tool_regex를 tools 배열에 추가하고, MCP 서버에 default_config: {defer_loading: true} 적용.
Phase 3 · LangChain Integration
create_agent 래퍼에서 Tool Search 자동 처리
LangChain create_agent 호출 시, 모델 공급사 및 선택 모드를 파악하여 Tool Search 설정을 자동 인젝션하는 래퍼 레이어를 구현했습니다. Agent 개발자(사용자)는 별도 코드 없이 Builder UI에서 토글만으로 Tool Search를 활성화할 수 있습니다.
Phase 4 · Verification
Claude API 로그 탭에서 입력 토큰 변화 실측
동일 요청(MCP 서버 2개, Tool ~80개 기준)에 대한 Tool Search 전·후 입력 토큰을 Claude API 로그 탭에서 비교 확인. Tool Search 활성화 후 입력 토큰 약 1/10 수준으로 감소 실측. GPT는 공식 메트릭 노출 인터페이스가 없어 간접 측정으로 대체했습니다.
3-2. 핵심 설정 코드 예시
📄

        "type": "toolsearch"    # Tool Search 활성화
    },
    {
        "type": "function",
        "function": { ... },

핵심 성과 및 기술 포인트

KEY RESULTS · TECHNICAL HIGHLIGHTS
📉 입력 토큰 절감

Claude 계열 Tool Search 활성화 후 API 로그 기준 입력 토큰 약 1/10 수준으로 감소. MCP 도구 포함 대규모 Tool 카탈로그에서 비용 및 응답 품질 동시 개선.

🔄 외부 MCP 동적 로딩

클라이언트에서 Schema를 직접 볼 수 없는 외부 MCP 서버 도구도 defer_loading: true로 LLM 서버 내부에서 동적 탐색·로딩 실현.

🎛 사용자 선택 모드

Claude 계열에서 BM25(시맨틱 자연어)와 Regex(패턴 정밀 매칭)를 Agent Builder UI에서 모드 전환으로 선택 가능. 사용 케이스에 따라 최적 전략 선택.

🤖 모델 자동 분기

GPT 계열(toolsearch type)과 Claude 계열(BM25/Regex)의 API 차이를 추상화. LangChain create_agent 래퍼가 모델에 따라 Tool Search 설정을 자동 분기·주입.
