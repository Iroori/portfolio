Portfolio · VDI Infrastructure Project

VM 상태 정보
수집 배치 시스템

Horizon REST API 기반으로 약 25,000대의 VM 상태를 주기적으로 수집하고 DB에 업데이트하는 배치 시스템 설계 및 구현. 이중화 호라이즌 제어, 병렬 처리, DB Lock 분리 전략을 포함한다.

C# / .NET Horizon REST API 병렬 처리 MSSQL Windows Scheduler VDI Infrastructure
01

프로젝트 배경

BACKGROUND · WHY THIS WAS NEEDED

문제 정의

VDI 포탈에서 VM의 실시간 상태(가동 여부, 사용자 연결 등)를 표시하려면 각 Horizon 서버로부터 VM 상태 정보를 주기적으로 수집해야 한다. 기존에는 Event DB 조회 방식을 사용했으나, 이는 실시간성이 낮고 이벤트 누락 가능성이 있었다. Horizon이 제공하는 REST API를 직접 호출하여 정확하고 최신 상태를 주기적으로 수집하는 배치 구조로 전환이 필요했다.

⚠
기존 방식의 한계
Event DB 폴링 방식으로 실시간 상태 반영이 느리고, 이벤트 누락 시 상태 불일치가 발생. 약 25,000대 VM의 상태를 커버하기에 구조적으로 부적합.
✓
개선 방향
Horizon REST API를 직접 호출하여 VM 상태를 pull 방식으로 수집. 배치 주기 5분(조정 가능) 목표, 병렬 처리로 수집 시간 최소화.
02

시스템 구조

SYSTEM ARCHITECTURE · OVERALL FLOW

수집 흐름 요약
Windows Scheduler → DB Horizon 테이블 조회 → 수집 대상 필터링 → 병렬 API 수집 → 별도 테이블 UPSERT
Step 01 — Trigger
Windows Task Scheduler 배치 기동
Windows Task Scheduler가 설정된 주기로 배치를 실행한다. 목표 주기는 5분이며, Horizon API 응답 속도와 DB 업데이트 처리 시간을 실측한 뒤 최종 확정할 예정이다.
Step 02 — Target Selection
수집 대상 Horizon 동적 조회
배치 시작 시 DB의 Horizon 테이블에서 collect_yn = 'Y'인 행만 조회하여 수집 대상 서버 목록을 구성한다. 이중화 구성(Active/Standby)된 호라이즌은 Standby 서버를 collect_yn = 'N'으로 설정해 중복 수집을 방지하며, DB 컬럼 변경만으로 즉시 대상 전환이 가능하다.
Step 03 — API Strategy
Horizon REST API 조회 전략 결정
포탈 조회 API를 참고하여 Pool 단위 필터 파라미터 지원 여부를 사전 조사한다. 필터가 지원되면 Pool별 분할 병렬 조회로 성능을 최적화하고, 지원되지 않으면 호라이즌 단위로 전체 일괄 조회 후 메모리 내에서 처리한다. 현장 담당자에게 필터 적용 vs 전체 조회의 응답 시간 차이 측정을 요청해야 한다.
Step 04 — Parallel Collection
호라이즌별 독립 스레드 병렬 수집
수집 대상 Horizon마다 독립 스레드를 할당해 병렬로 API를 호출한다. 가장 많은 호라이즌 기준 약 8,000대 VM이 단일 스레드에서 처리되며, API가 필터를 지원할 경우 Pool 단위로 추가 분할해 서브 태스크로 병렬화한다. 전체 수집 대상은 약 25,000대이다.
Step 05 — DB Write
별도 상태 테이블에 UPSERT
수집한 VM 상태를 기존 운영 테이블이 아닌 배치 전용 별도 VM 상태 테이블에 UPSERT한다. 포탈 및 Trinity의 조회 쿼리·프로시저는 이 테이블을 JOIN 시 WITH (NOLOCK) 옵션을 적용하여 Lock 경합을 우회한다.
03

핵심 설계 결정

KEY DESIGN DECISIONS · 4 ISSUES

🔀
이중화 호라이즌 수집 제어
Horizon은 이중화(Active/Standby)로 구성된 서버가 존재한다. 모든 호라이즌에서 수집하면 동일 VM의 상태가 중복 수집되는 문제가 발생한다. 이를 막기 위해 DB Horizon 테이블에 collect_yn CHAR(1) 컬럼을 추가하고, 배치는 해당 컬럼이 'Y'인 서버만 대상으로 삼는다. 이중화 전환 시에도 DB 값 변경만으로 즉시 대상을 바꿀 수 있어 운영 유연성이 높다.
🔒
DB Table Lock 분리 전략
VM 상태는 5분마다 지속적으로 업데이트되므로, 기존 운영 테이블에 직접 쓰면 포탈 SELECT와 배치 UPDATE 간 Lock 경합이 발생할 수 있다. 이를 방지하기 위해 배치 전용 별도 상태 테이블(vm_status_batch 등)을 별도 생성한다. 포탈·Trinity에서 VM 상태를 조회하는 모든 View, Procedure, Query는 이 테이블 JOIN 시 WITH (NOLOCK) 옵션을 일괄 적용한다. 기존 프로시저의 NOLOCK 패턴을 참고하여 동일한 방식으로 작성한다.
🔍
REST API 조회 전략 (사전 조사 필요)
포탈에서 사용하는 Horizon REST API를 참고하여 Pool 단위 필터 파라미터 지원 여부를 확인한다.

Case A — 필터 지원 Pool 별로 필터를 걸어 분할 병렬 조회. 성능 최적화 가능.
Case B — 필터 미지원 호라이즌 단위로 전체 일괄 조회 후 메모리 내 처리.

어느 경우든 현장 담당자에게 필터 적용 vs 전체 조회 응답 시간 비교 측정을 요청해야 하며, 결과에 따라 전략을 확정한다.
⚡
배치 주기 조율
목표 주기는 5분이나, 실제 소요 시간은 API 응답 속도·필터 지원 여부· DB UPSERT 처리 속도에 따라 달라진다. 개발 환경에서 전체 25,000대 수집 소요 시간을 실측하고, 병렬 스레드 수 조합을 테스트하여 최적 구성을 확정한 뒤 배치 주기를 최종 결정한다.

06

기술 스택

TECH STACK · IMPLEMENTATION

💻
배치 프로그램 — C# / .NET
Windows Task Scheduler로 기동되는 C# 콘솔 애플리케이션. Task.WhenAll 또는 Parallel 클래스를 활용해 Horizon별 독립 스레드를 구성하고 병렬로 API를 호출한다.
🌐
Horizon REST API
VMware Horizon이 제공하는 REST API를 HttpClient로 호출. 포탈 조회 API와 동일한 엔드포인트를 참고하며, Pool 필터 파라미터 지원 여부에 따라 분할 조회 또는 일괄 조회 방식을 결정한다.
🗄
MSSQL · 별도 상태 테이블
수집 결과는 배치 전용 VM 상태 테이블에 UPSERT(MERGE 또는 INSERT … ON CONFLICT 패턴). Trinity 및 포탈의 조회 Stored Procedure · View에는 WITH (NOLOCK)을 적용하여 배치 Write와의 Lock 경합을 방지한다.
