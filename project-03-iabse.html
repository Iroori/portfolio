01

AWS Lightsail 인프라 구축

AWS LIGHTSAIL · NGINX · SQL SERVER · SYSTEMD · LET'S ENCRYPT
🎯 인프라 설계 근거

약 800명 규모 이벤트성 컨퍼런스 사이트는 트래픽 피크가 짧고 예측 가능합니다. EC2(가변 과금) 대비 Lightsail(4GB RAM, 80GB SSD, 월 고정 $20)은 비용 예측 가능성이 높고 네트워킹 설정이 단순합니다. SQL Server 2022 Express를 동일 인스턴스에 올려 포트를 외부에 노출하지 않고 localhost:1433으로 통신해 방화벽 복잡도를 최소화했습니다.

1-1. 서버 구성 스택
☁️
AWS Lightsail
ap-northeast-2 (서울) · Ubuntu 22.04 · 4GB RAM · 80GB SSD · 고정 IP 할당 · Route 53 도메인 연결
⚡
Nginx 리버스 프록시
React 빌드 정적 서빙 · /api → Spring Boot 프록시 · gzip · Cache-Control 헤더
🗄
SQL Server 2022 Express
동일 인스턴스 localhost 통신 · prod/dev DB 분리 · 외부 포트 미노출
🔒
Let's Encrypt HTTPS
Certbot 자동 갱신 · HTTP → HTTPS 리다이렉트 · systemd 서비스로 Spring Boot 관리
1-2. Nginx 캐시·압축 전략 (실측 확인)

실측 결과 (curl -sI): Vite 빌드 산출물(index-EVOWYLeG.js 등 content-hash 파일명)에 Cache-Control: max-age=31536000, public, immutable + Expires 헤더 적용 확인. 파일명에 해시가 포함되므로 배포 시 자동 무효화, 캐시 오염 없음. HTML 응답에는 Content-Encoding: gzip 압축 적용.

1-3. AWS CLI — 임시키 기반 DB 접근
🔑
Lightsail 임시 SSH 키 (1분 만료) 활용
초기 인프라 구축 시 aws lightsail get-instance-access-details CLI 명령으로 1분 만료 임시 SSH 키를 발급받아 서버에 접근했습니다. extract_ssh.py로 JSON 응답에서 키를 파싱 → 즉시 sqlcmd로 DB 마이그레이션 실행 → 키 자동 만료. 장기 SSH 키를 로컬에 보관하지 않는 보안상 이점이 있습니다.
Trade-off
단일 인스턴스 SQL Server 공존 한계

JVM(Spring Boot)과 SQL Server 2022 Express가 4GB 메모리를 공유합니다. Express 버전의 DB 크기 상한(10GB)과 RAM 사용 상한(1.4GB)이 존재합니다. 단일 장애점(서버 장애 시 DB도 함께 중단) 구조이며, 개별 스케일링이 불가능합니다.
판단 근거
이벤트 규모 대비 적합한 트레이드오프

800명 이벤트성 사이트의 DB 크기는 수십 MB 수준으로 10GB 상한과 무관합니다. 동시 접속 최대치도 낮고 행사 기간이 한정적입니다. RDS 분리 시 추가 비용($50+/월)과 복잡성 대비 이점이 없다고 판단해 비용·단순성 우선 설계를 선택했습니다. 향후 부하 증가 시 RDS 분리 또는 Standard 에디션 업그레이드 경로가 열려 있습니다.

02

GitHub Actions — 자동 배포 파이프라인

GITHUB ACTIONS · CI/CD · ZERO-CLICK DEPLOY
🎯 설계 목표

main 브랜치에 push하면 Maven 빌드 → Vite 빌드 → 서버 파일 전송 → 서비스 재시작 → 헬스체크까지 사람 개입 없이 자동으로 완료되는 파이프라인을 구성했습니다. PG MID·SSH 키·DB 접속 정보 등 민감값은 GitHub Secrets에서 주입해 소스코드에 노출되지 않습니다.

Step 01 / Backend Build
Java 17 + Maven 빌드 (테스트 스킵)
actions/setup-java@v4로 Temurin 17 설정 후 ./mvnw clean package -DskipTests -q로 실행 가능한 JAR 생성.
Step 02 / Frontend Build
Node 20 + Vite 빌드 (Secrets 주입)
package-lock.json 기반 npm cache로 빌드 속도 최적화. PG 가맹점 ID(VITE_PAYGATE_MID_DOMESTIC, _OVERSEAS)를 GitHub Secrets에서 환경변수로 주입 후 npm run build — 소스코드에 MID 하드코딩 없음.
Step 03 / File Transfer
SCP + rsync 분리 전송
백엔드 JAR는 SCP로 app.jar.new로 먼저 전송 후 교체(atomic swap). 프론트엔드는 rsync --delete로 삭제·추가만 전송해 전송량 최소화. SSH 키는 Secrets에서 런타임에 chmod 600 설정 후 즉시 사용.
Step 04 / Deploy + Health Check
무중단에 가까운 재시작 + 자동 롤백 방어
systemctl stop → app.jar.new → app.jar 교체 → systemctl start 순서로 교체 전 old JAR이 계속 서빙됩니다. 기동 후 GET /api/health를 2초 간격 최대 60회(120초) 시도. 헬스체크 타임아웃 시 journalctl 로그 30줄 출력 + exit 1로 Actions 실패 표시.

Secrets 관리 범위: LIGHTSAIL_HOST, LIGHTSAIL_SSH_KEY, VITE_PAYGATE_MID_DOMESTIC, VITE_PAYGATE_MID_OVERSEAS, VITE_PAYGATE_METHOD — 총 5개 Secrets. 빌드 로그에도 마스킹 처리되어 노출 없음.

GitHub Actions Maven Build Vite Build SCP / rsync Health Check Secrets 관리

보안 아키텍처 설계

JWT · BCRYPT · SHA-256 · TOKEN ROTATION
3-1. 비밀번호 이중 해싱 — 네트워크 평문 노출 차단
🔐
클라이언트 SHA-256 → 서버 BCrypt 이중 해싱
클라이언트는 평문 비밀번호를 crypto.subtle.digest('SHA-256')으로 해싱해 64자 hex 문자열만 네트워크로 전송합니다. 서버는 이 SHA-256 hex를 BCryptPasswordEncoder로 다시 해싱해 DB에 저장합니다.

결과적으로 네트워크 탭에 평문 비밀번호가 절대 노출되지 않으며, 서버에서도 평문을 보지 않습니다. 백엔드 SignupRequest.password는 @Size(min=64, max=64)로 SHA-256 hex 길이를 검증합니다.

Client → SHA-256("Test1234!") → "a665a459..." (64자) → 서버 BCrypt → "$2a$10$..." → DB

3-2. JWT Access + Refresh Token Rotation
인증 흐름
Stateless JWT + DB Refresh Token
로그인 → Access JWT(단기) + Refresh Token(UUID, DB 저장). Access 만료 시 POST /api/auth/refresh로 Token Rotation. 로그아웃 시 DB에서 Refresh Token 삭제.
JwtAuthFilter 최적화
Filter에서 DB 조회 없이 클레임 기반 인증
JwtAuthFilter는 JWT 클레임에서 직접 인증 객체를 생성해 매 요청마다 DB 조회를 하지 않습니다. UserDetailsService는 로그인(AuthenticationManager) 시에만 호출됩니다. 요청 처리 성능과 보안을 동시에 확보합니다.
운영 안전장치
dev-mode 플래그 + SecurityAuditService
app.dev-mode=false 시 /api/dev/** 엔드포인트가 자동 비활성화되어 운영 환경에서 개발용 API가 노출되지 않습니다. SecurityAuditService는 이메일 등 민감 정보를 자동 마스킹해 감사 로그를 기록합니다.
3-3. 환경별 완전한 코드·설정 분리
🏠
dev 프로파일 (H2/MSSQL + console 메일)
ddl-auto: update · 시드 계정 자동 생성 · 이메일 코드 콘솔 출력 · /api/dev/** 활성화
🏢
prod 프로파일 (SQL Server + SES/SMTP)
ddl-auto: validate · 모든 민감값 환경변수 주입 · dev API 비활성화 · show-sql: false
SHA-256 + BCrypt JWT Access Token Refresh Token Rotation dev-mode 격리 Security Audit

글로벌 결제 시스템 (PayGate) 연동

PAYGATE · DYNAMIC MID ROUTING · DUAL VERIFICATION
🎯 구축 목표

국내·해외 참가자가 공존하는 국제 컨퍼런스 특성상, 사용자 국가 코드에 따라 국내용/해외용 가맹점 MID를 자동 분기하고, PG 결과를 서버사이드에서 이중 검증해 안전한 트랜잭션 흐름을 구축했습니다.

💳
Dynamic MID 분기 라우팅 (국가별)
React 단에서 국가 Select 입력을 트래킹해 결제 버튼(PGIOForm) Submit 시 국내용(VITE_PAYGATE_MID_DOMESTIC)·해외용(VITE_PAYGATE_MID_OVERSEAS) MID를 동적으로 맵핑해 PayGate에 전송합니다. MID는 빌드 타임에 Secrets에서 주입되어 소스에 노출되지 않습니다.
🔗
OpenPayAPI 서버사이드 이중 검증
Step3Payment 컴포넌트에서 OpenPayAPI 콜백(getPGIOresult) 수신 후, 결과값을 Node.js 백엔드로 전달해 PayGate 검증 API(verifyReceived.jsp)로 서버사이드 재검증합니다. 클라이언트 결과만 믿지 않는 이중 무결성 구조입니다.
📅
등록 기간 기반 가격 자동 분기
사전등록(Pre) / 얼리버드(Early-bird) / 정규(Regular) 3단계 기간을 Config(application.yaml)에서 환경변수로 주입합니다. 결제 요청 시 서버가 현재 날짜를 기준으로 적용 가격 구간을 자동 판단합니다. 기간 변경 시 배포 없이 환경변수만 수정하면 됩니다.
05

기간별 부분 환불 자동화

PARTIAL REFUND · D-DAY ALGORITHM · ACID TRANSACTION

행사 특성상 취소 시점에 따라 위약금 수수료율이 달라집니다. 기존 관리자 수동 계산·처리를 스케줄러 기반 완전 자동화로 전환했습니다.

Step 01 / Date Calculation
행사 D-Day 역산 위약금 알고리즘
환불 요청 타임스탬프와 행사 메인 이벤트 Date(PAYGATE_EVENT_DATE)를 비교해 D-90(100% 환불) / D-30(50% 환불) 등 구간별 수수료율을 자동 계산. 이벤트 날짜도 환경변수로 주입되어 배포 없이 변경 가능합니다.
Step 02 / PayGate Cancel
PayGate Cancel API — 부분 취소액 전송
원결제 TID를 참조해 수수료가 차감된 부분 취소액(Partial Amount)을 산출하고 PayGate 취소 API(cancelAPI.json)에 POST 요청을 전송합니다.
Step 03 / ACID Transaction
PG 취소 확인 후 DB 상태 업데이트
PG사에서 Success 응답이 내려온 후에만 DB 상태값을 업데이트하고 환불 이력을 영구 기록합니다. @Transactional 보장으로 PG 취소·DB 갱신 간 원자성을 확보합니다.
환경별 스케줄러 제어
prod 환경에서만 Cleanup Scheduler 활성화
process.env.NODE_ENV / app.dev-mode 확인으로 미완료 더미 데이터 정리 Cron이 개발 환경에서 돌지 않도록 제약. dev/prod 경합 장애를 사전 차단합니다.
D-Day 역산 알고리즘 PayGate Cancel API ACID Transaction Cron Scheduler

이메일 발송 추상화 — Provider 전환 구조

MAIL ABSTRACTION · SES · GMAIL SMTP · PROVIDER PATTERN
🎯 설계 배경

AWS SES 프로덕션 발송 권한 취득 전까지 Gmail SMTP를 임시 대체 수단으로 활용합니다. Provider 교체 시 코드 수정 없이 Config 값 하나(app.mail.provider)만 변경하면 되도록 MailService 인터페이스 + @ConditionalOnProperty 구현체 패턴으로 추상화했습니다.

📧
MailService 인터페이스 + 3개 구현체
ConsoleMailService — dev: 콘솔 로그 출력 (SMTP 불필요)
GmailMailService — SES 인증 전 임시: Gmail SMTP
SesMailService — prod: AWS SES SMTP (DKIM 인증 완료)

application.yaml의 app.mail.provider: ses|gmail|console 값에 따라 @ConditionalOnProperty로 구현체가 자동 선택됩니다. 배포 환경 변경 없이 Config 재시작만으로 전환됩니다.

현재 상태: AWS SES DKIM 인증은 완료됐으나 프로덕션 발송 권한 신청 중. Gmail SMTP로 운영 중이며, SES 승인 완료 시 app.mail.provider=ses 변경만으로 전환됩니다.

AWS SES Gmail SMTP @ConditionalOnProperty MailService 추상화 DKIM 인증
